Cloud storage systems have long struggled with balancing replication
overhead and read latency. Existing approaches either replicate
aggressively (causing high write amplification) or rely on erasure coding
(causing high read latency on degraded paths). We introduce ChainSync,
the first storage system to dynamically adjust replication factor based
on observed access patterns at the file level.

ChainSync monitors a sliding window of read frequency for each file and
migrates between replication factor 2, 3, and erasure-coded states based
on a learned threshold policy. Files that exceed the access threshold
are upgraded to higher replication; cold files are demoted. The
migration is performed asynchronously to avoid impacting foreground I/O.

Our evaluation on a 12-node cluster shows that ChainSync reduces aggregate
storage cost by 31% compared to fixed replication factor 3, while keeping
p99 read latency within 4ms of the replicated baseline. This demonstrates
that adaptive replication is a practical alternative to static policies.

We also show that ChainSync's policy is robust to workload shifts: when
we transition from a YCSB-A workload to YCSB-C mid-experiment, the
policy converges to the new optimal within 12 minutes. This is the first
demonstration of online policy adaptation for storage replication. Our
results, taken together, demonstrate that learned tiering policies are
deployment-ready for production-scale clusters and offer a clean break
from the static-policy era of distributed storage.

## Revised text (conservative rewrite)

Cloud storage systems face a tension between replication overhead and read latency on degraded paths: full replication inflates storage cost, while erasure coding raises tail latency when reconstruction is required. Most deployed systems resolve this tension with a static, system-wide policy. We present ChainSync, a storage system that adjusts per-file redundancy — between 2x replication, 3x replication, and an erasure-coded state — in response to observed read frequency.

ChainSync maintains a sliding-window read-frequency estimate per file and migrates files across redundancy states using a threshold policy. Hot files are promoted to higher replication; cold files are demoted to erasure coding. Migrations run asynchronously off the foreground I/O path.

On a 12-node cluster running [workload mix to be specified], ChainSync reduces aggregate storage cost by 31% relative to a uniform 3x replication baseline, while p99 read latency stays within 4 ms of that baseline. These results suggest that per-file adaptive redundancy can recover much of the cost advantage of erasure coding without the tail-latency penalty on the workloads we tested. Broader claims about production readiness will require evaluation at larger scale, under failure conditions, and across more diverse workloads.

## Issues I would not let pass in review

**1. "First" claims are unsupported and probably false.**
- "the first storage system to dynamically adjust replication factor based on observed access patterns" — adaptive/tiered redundancy has a substantial prior literature (e.g., EAR, HACFS, AutoRAID-style tiering, Facebook's f4 warm-tier transitions, Ceph's CRUSH-driven tiering, and various academic systems on hot/cold-aware EC). At minimum this needs a related-work pass and the claim should be narrowed (e.g., "to our knowledge, the first to do X at file granularity online with a learned threshold").
- "first demonstration of online policy adaptation for storage replication" — same problem. Drop or scope tightly.

**2. "Learned threshold policy" is asserted but not described.**
The abstract calls the policy "learned" and later "learned tiering," but the mechanism description only says "threshold policy." Either:
- it is a hand-tuned threshold (then stop calling it learned), or
- it is actually learned (then say from what signal, what objective, what training/online procedure).
This inconsistency is the single biggest credibility issue in the text.

**3. The 31% / 4 ms result needs context to be interpretable.**
- 31% vs. 3x replication is the easy comparison. The harder and more honest comparison is vs. erasure coding alone, or vs. a static hot/cold tiering baseline. Without those, the cost win is not attributable to *adaptivity*.
- "p99 within 4 ms" — 4 ms of what absolute number? 4 ms on a 2 ms baseline is catastrophic; 4 ms on a 40 ms baseline is noise. Report absolute latencies.
- Single 12-node cluster, workload not fully specified, no failure-mode results, no write-latency numbers, no migration-bandwidth cost.
